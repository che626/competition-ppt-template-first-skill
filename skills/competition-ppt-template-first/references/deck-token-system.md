# Deck Token System

Use `templates/deck-style-tokens.md` before page-template production. A deck token system is not a web-design exercise. It is a compact control surface that keeps a competition deck visually coherent while allowing page compositions to vary.

## What to lock once per deck

1. **Color roles**: canvas, primary ink, secondary ink, primary accent, support accent, and warning signal. Do not let every slide introduce a new arbitrary color.
2. **Type roles**: display, title, claim, section, body, and caption. Define a practical minimum body size; do not solve overflow by creating a smaller unofficial size.
3. **Spatial roles**: edge margin, group gap, inner gap, shared title origin, and caption offset.
4. **Image roles**: how real evidence, screenshots, template surfaces, and decorative cutouts are framed and captioned.
5. **Contrast rhythm**: which pages are rich-scene, mid-tone, or quiet-evidence. A visual family is not a deck of identical backgrounds.

## Use semantic roles, not raw decisions

Say `accent-primary` because it marks the one judge-relevant conclusion; do not say "make this random line yellow." Say `caption` because it labels a real artifact; do not make captions accidentally compete with body text.

## Change discipline

Change a token only when the subject, user feedback, or readability requires it. Record the change and affected slides. Do not patch a single weak page with a new unrelated color, font, border, or shadow.

## Slide-level decision

Each page blueprint names the token version and the intentional exceptions. An exception is allowed only when it strengthens the page role, for example a document-readable light page in an otherwise dark product deck. It must still use the same typography and spacing logic.
