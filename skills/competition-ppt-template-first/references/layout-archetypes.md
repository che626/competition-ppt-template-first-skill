# Layout Archetypes for Competition Defense Decks

Use a deck-level visual system, but vary page structure. Select the archetype from the page's proof burden, not from habit.

## 1. Cover: hero scene + identity

Best for: first page and conclusion.

- One cinematic scene that belongs to the project domain.
- Large project name.
- Minimal secondary metadata.
- A small amount of room to reveal the next visual beat.

Avoid: dashboard-style cards, excessive tag chips, fake dense text.

## 2. Pain-point: three failures around a visual scene

Best for: industry problem, user problem, or current limitation.

- One relevant scene or artifact in the center or background.
- Three asymmetrical evidence clusters, each with a clear failure mode.
- A strong bottom conclusion that reframes the problem.

Avoid: three identical rounded cards with generic icons.

## 3. Solution: central visual transformation

Best for: design scheme or technical logic.

- Center: the real visual result, input image, or product proof.
- Left: input/problem/visual perception layer.
- Right: structured result, analysis, or downstream value.
- Bottom: the closed loop in one line or short sequence.

The page should show transformation, not a feature list.

## 4. Model choice: decision matrix + deployment evidence

Best for: model selection and technical credibility.

- Main area: 2-4 models compared on verified criteria.
- Supporting area: actual inference result, latency, architecture, or deployment path.
- Highlight the chosen model through evidence, not exaggerated color.

Never claim benchmark numbers that are not in the report.

## 5. Iteration: version story

Best for: model or system evolution.

- Three version states along a visual path.
- Each state uses one real artifact and one concise improvement claim.
- One bottom strip explains what the iteration unlocks.

Avoid: an empty arrow between three identical rectangles.

## 6. System showcase: product screenshots as the anchor

Best for: product capability, feature groups, management/configuration pages.

- Screenshots occupy the largest share of the slide.
- Labels explain what judges should notice in each screenshot.
- Use a restrained template because screenshots already provide density.

Do not use generated UI screenshots when real screens exist.

## 7. Evidence wall: artifact-led proof

Best for: papers, software copyrights, patents, awards, datasets, test records.

- Group real artifacts by proof purpose, not just file type.
- Reserve larger readable frames for certificates and documents.
- Use a small right-side argument column to explain why the evidence matters.

When the card geometry is wrong, regenerate the entire background template instead of shifting certificate pictures over a stale underlay.

## 8. Roadmap: grounded next steps

Best for: future planning.

- Sequence stages or capability layers.
- Clearly distinguish completed work, work in progress, and future goals.
- Use the project domain to make each milestone concrete.

Avoid: a generic five-box timeline with no real operational meaning.
